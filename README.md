# 3. История браузера (Browser History)

## Описание задачи

Необходимо реализовать систему хранения истории браузера. Каждая запись содержит:

- URL  
- время посещения  
- флаг "закладка"  

## Обязательная часть

Реализовать:

- Навигацию «назад / вперед»  
- Очистку истории  
- Поиск по домену  

## Вариативная часть (выполнены задания 1 и 2)

Реализованы следующие дополнительные функции:

1. Сохранение истории в файл в формате Base64 (одна строка — одна запись)  
2. Загрузка истории из файла в формате Base64  


# Требования к выполнению

- Использовать линейные структуры данных (динамический массив или LinkedList)  
- Обосновать выбор структуры данных  
- Обработать пограничные случаи (пустые списки, выход за границы и т. д.)  
- Код должен быть читабельным и структурированным  
- Оформить репозиторий с кодом и выложить на github  

## Обоснование выбора структуры данных

В проекте использован **динамический массив (срез, slice)** вместо связного списка (LinkedList) по следующим причинам:

- **Произвольный доступ по индексу за O(1)** – для навигации «назад/вперёд» требуется быстро получать запись по текущей позиции. В связном списке для этого пришлось бы каждый раз проходить от начала или конца, что даёт O(n) на шаг.
- **Эффективное использование памяти** – срез хранит элементы последовательно в памяти, что уменьшает накладные расходы на указатели (в отличие от LinkedList, где каждый узел требует дополнительной памяти для ссылок).
- **Простота реализации** – в Go срезы являются идиоматичной структурой, подходящей для большинства задач с последовательным хранением данных. Операции добавления в конец (амортизированное O(1)), обрезания хвоста (при добавлении новой записи) и очистки реализуются лаконично.
- **Лучшая локальность кэша** – последовательное хранение улучшает производительность при итерациях (поиск по домену, сохранение/загрузка в файл).

Связный список дал бы преимущество только при частых вставках в середину, но в истории браузера новые записи всегда добавляются в конец после обрезания «будущих» страниц, а удаление из середины не требуется. Таким образом, динамический массив является оптимальным выбором.
